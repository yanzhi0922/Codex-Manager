## P0

### 1. 把会话标识派生收敛成单一入口
- 目标：`conversation_id`、`session_id`、`x-client-request-id` 的派生逻辑只保留一个共享函数。
- 现状：`local_validation`、`transport`、`openai_fallback` 各自处理，容易出现语义漂移。
- 验收：
  - 同一输入在三条链路里得到一致结果。
  - 新增/修改一个字段时，不需要多处同步改逻辑。

### 2. 把 `turn_state` 规则单独抽成显式策略
- 目标：明确哪些条件下保留、哪些条件下清理 `x-codex-turn-state`。
- 现状：当前是“线程锚点一致则保留，否则清理”的隐式判断，后续很容易被别的分支打散。
- 验收：
  - 规则有单独函数或枚举表达。
  - `transport` 和 `openai_fallback` 共用同一套判断。

### 3. 把 `prompt_cache_key` 的重写路径做成可追踪的单点
- 目标：清楚区分“初始注入”“强制覆盖”“回退补写”三种来源。
- 现状：请求改写和上游发送两侧都在碰它。
- 验收：
  - 日志里能看出最终值来自哪一层。
  - 不同协议的重写策略可独立测试。

## P1

### 4. 增加请求头快照测试
- 目标：直接锁住关键请求头的最终形态。
- 建议覆盖：
  - `conversation_id` 存在
  - `conversation_id` 缺失
  - `prompt_cache_key` 与 `conversation_id` 一致
  - `prompt_cache_key` 与 `conversation_id` 冲突
- 验收：
  - 能通过测试证明 `x-client-request-id`、`session_id`、`x-codex-turn-state` 的最终输出。

### 5. 补一条“线程锚点冲突”可观测日志
- 目标：当 `conversation_id` 与 `prompt_cache_key` 不一致时，能在日志里直接看出来。
- 现状：虽然逻辑已清理，但排障时还需要回看代码推导。
- 验收：
  - 冲突场景有稳定日志键值。
  - 不影响正常路径的日志噪声。

### 6. 把请求对齐文档和代码保持同源
- 目标：避免“代码已改、对照表没改”的情况再次发生。
- 建议：
  - 每次改请求语义时同步更新 `docs/report` 下的差异表。
  - 文档里注明当前版本对应的代码文件。

## P2

### 7. 路由与轮转逻辑再解耦一点
- 目标：账号轮转只看路由层状态，不被请求头派生细节影响。
- 现状：已通过 route plan 进一步把“是否应用 route strategy”与“候选池旋转”分开。
- 验收：
  - 路由层只消费明确的 affinity 结果。
  - 请求头修改不会影响轮转策略本身。

### 8. 把异常账号态和可用性判断拆开
- 目标：区分“token 还能换”与“账号已停用/不可用”。
- 现状：`usage endpoint`、token 交换、后续请求错误码应该分层解释。
- 验收：
  - `account_deactivated`、`401`、`403`、`429` 有独立处理建议。
  - 不再把 token 交换成功误判为账号可用。
